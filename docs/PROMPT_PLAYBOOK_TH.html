Prompt Framework Guide · Visual Edition

เขียน Prompt ให้ AI เข้าใจงานตั้งแต่รอบแรก

คู่มือนี้รวม RTF, RACE, CARE, CREATE, STAR และ PAR จากเอกสารสอนเดิม พร้อมตัวอย่างที่ใช้กับ Portfolio Template จริง โดยลดเนื้อหาซ้ำและอธิบายว่า “แต่ละช่องมีไว้ทำอะไร” มากกว่าการให้จำตัวย่ออย่างเดียว

เลือก Frameworkเรียนทีละสูตรก่อน–หลัง4 Prompts สร้างโปรเจกต์แบบฝึกหัด

1. Prompt ที่ดีมีหน้าที่เหมือน “ใบสั่งงาน”

บอกว่า AI เป็นใคร
→
บอกว่าต้องทำอะไร
→
ให้บริบทที่พอ
→
กำหนดผลลัพธ์
→
ใส่ข้อห้าม
หัวใจสำคัญ: Prompt ที่ดีไม่จำเป็นต้องยาว แต่ต้องลดพื้นที่ให้ AI เดาเอง โดยเฉพาะงานโค้ด งานเอกสาร และงานที่มีข้อจำกัด

2. เลือก Framework จากประเภทงาน

RTFงานเร็ว
RACEมีบริบท
CAREมีตัวอย่าง
CREATEงานใหญ่
STARวิเคราะห์ปัญหา
PARแก้เป็นชั้น
สถานการณ์สูตรที่เหมาะเหตุผล
สรุปบทความเป็นตารางRTFงานสั้นและรู้รูปแบบผลลัพธ์
ออกแบบเว็บราชการให้ทันสมัยแต่สุภาพRACE / CAREต้องคุมบริบท โทน และ reference
สร้างระบบ Portfolio ทั้งโปรเจกต์CREATEมีหลายส่วน หลายข้อห้าม และหลายไฟล์
Mobile Coverflow ไม่ยุบปิดSTAR / PARต้องแยกอาการ สาเหตุ วิธีแก้ และผลลัพธ์

3. Framework แต่ละแบบ

RTF — Role · Task · Format

Role = ให้ AI เป็นใครTask = ให้ทำอะไรFormat = ตอบแบบไหน

เหมาะกับผู้เริ่มต้นและงานที่ scope ไม่ใหญ่

Role:
You are a senior frontend developer.

Task:
Explain CSS Grid and Flexbox for beginners.

Format:
Return a comparison table with short examples.
ถ้าขาด Format: AI อาจตอบยาวหรือจัดรูปแบบไม่เหมาะกับการใช้งาน

RACE — Role · Action · Context · Expectation

RoleActionContextExpectation

เหมาะกับงานองค์กร UX/UI และงานที่ต้องควบคุมคุณภาพ

Role:
You are an enterprise UX/UI designer.

Action:
Improve the Portfolio Mode hierarchy.

Context:
The product supports Portfolio, Resume, themes, TH/EN, print and mobile coverflow.

Expectation:
Keep it professional. Do not remove controls or break mobile.

CARE — Context · Action · Result · Example

ContextActionResultExample

เหมาะกับงานที่มีภาพอ้างอิง งานออกแบบ งานเขียน และงาน Canva

Context:
This is a professional portfolio website.

Action:
Create the hero visual direction.

Result:
Premium, calm, trustworthy and modern.

Example:
Use the feeling of a high-end SaaS landing page, not a gaming poster.
อย่าเพียงส่งรูป: ควรบอกด้วยว่าชอบ “อะไร” ในตัวอย่าง เช่น spacing, mood, typography หรือ interaction

CREATE — Context · Request · Explanation · Action · Task · Extras

เหมาะกับ feature ใหญ่ refactor หรือการเริ่มโปรเจกต์ใหม่

ภาพรวมคำขอหลักเหตุผลขั้นตอนงานย่อยข้อห้าม
Context:
[ระบบปัจจุบันและผู้ใช้]

Request:
[งานหลัก]

Explanation:
[เหตุผลและ behavior]

Action:
1. Inspect
2. Plan
3. Implement
4. Verify

Task:
- [งานย่อย]

Extras:
- Preserve [ของเดิม]
- Do not [ข้อห้าม]
- Report risks and checks

STAR — Situation · Task · Action · Result

เหมาะกับ debug, root cause และ optimization

Situation:
On mobile, images move but the card does not close at the end.

Task:
Fix mobile collapse without breaking desktop.

Action:
Inspect progress logic, media queries and class timing.

Result:
Return root cause, minimal patch and regression checklist.
ประโยชน์: ช่วยแยก “อาการที่เห็น” ออกจาก “สิ่งที่ต้องแก้” ทำให้ AI ไม่รีบแก้ผิดจุด

PAR — Problem · Action · Result

เหมาะกับ bug เฉพาะจุด งานแบบแบ่งเลเยอร์ และ AI ที่มี context จำกัด

Problem:
Mobile coverflow stays open after the last image.

Action:
Add open–move–hold–close phases using the same progress model as desktop.

Result:
The image area closes and the next card follows without blank rail.

4. เปรียบเทียบ Prompt ก่อนและหลัง

ก่อน: กว้างและเดาเยอะ

เมนูมือถือยังซ้ายอยู่ แก้ให้หน่อย

AI ยังไม่รู้ viewport, selector, สิ่งที่ห้ามกระทบ หรือผลลัพธ์ที่ตรวจได้

หลัง: ตรวจได้และปลอดภัยกว่า

Problem:
On mobile, the nav container is centered but the controls remain left aligned.

Action:
Inspect computed width, justify-content, overflow, stylesheet order and inline styles. Identify the winning rule before editing.

Result:
Controls are centered above 360px. Narrower screens may scroll without clipping.

5. สูตรป้องกัน AI แก้งานเกิน Scope

Scope

ระบุไฟล์หรือ section ที่อนุญาตให้แตะ

Preserve

บอกว่าฟังก์ชันใดต้องไม่เสีย

Do not

ห้าม rewrite, เปลี่ยน schema หรือเพิ่ม dependency โดยพลการ

Acceptance criteria

กำหนดพฤติกรรมที่ตรวจได้

Test matrix

Desktop, Mobile, Theme, Print และ Export

Report

Changed, Preserved, Risk, Verified, Not verified

6. คัมภีร์ 4 Prompts สร้าง Portfolio แบบแบ่งเลเยอร์

เนื้อหาเดิมถูกปรับให้ตรงกับระบบปัจจุบัน: ไม่ใช้ horizontal rail บน mobile และให้ Coverflow responsive เหมือน Desktop

Prompt 1 — Data Layer และ Shell
Problem:
Create a static bilingual Portfolio/Resume generator without a database.

Action:
- Define schema for profile, contact, skills, projects, theme, layout and language.
- Store data in localStorage with schemaVersion.
- Add navigation, language, layout, theme and edit controls.
- Escape user-provided text.

Result:
A safe bilingual shell that stores data in the browser and uses non-sensitive demo data.
Prompt 2 — Theme Engine และ Components
Problem:
Portfolio and Resume must share the same visual identity across themes and dark/light modes.

Action:
- Create central theme tokens.
- Make Header, Bio, Skills, Contact, Experience and Resume consume the same tokens.
- Separate screen tokens and print tokens.
- Avoid fixed colors that cause Resume mismatch.

Result:
All modes stay readable and visually consistent.
Prompt 3 — Responsive Coverflow Scrollytelling
Problem:
Horizontal image rails create trailing blank space and poor mobile behavior.

Action:
- Use static layout for one image.
- Use vertical-scroll coverflow for two or more images.
- Use requestAnimationFrame.
- Apply open 0.06, move 0.12–0.76, hold, close 0.90.
- Use responsive step, angle and scale on mobile.
- Flatten images for print.

Result:
Smooth coverflow on desktop and mobile with no blank rail.
Prompt 4 — Mobile, Print, Export และ Deploy
Problem:
Mobile controls are difficult to tap and print/export may hide content due to animation state.

Action:
- Make navigation safe-area aware with practical touch targets.
- Force print content visible and hide interactive controls.
- Keep public and dist synchronized.
- Load assets through prepare-vercel with cache versions.
- Verify the deployed production path.

Result:
Mobile is usable, print content is complete and production loads the new assets.

สรุปสูตรจำง่าย

RTF = เร็ว · RACE = มีบริบท · CARE = มีตัวอย่าง · CREATE = งานใหญ่ · STAR = วิเคราะห์ปัญหา · PAR = แก้เป็นชั้น

ไม่ต้องใช้ทุกช่องทุกครั้ง เลือกเท่าที่จำเป็น แต่สำหรับงานโค้ดควรมี Scope, Preserve, Do not และ Acceptance criteria เสมอ
